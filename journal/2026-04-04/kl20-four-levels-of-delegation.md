## 1. 核心发现

从 OpenClaw 文档中研究发现，自动化不是"设置定时任务"，是**逐步转移决策权**的过程。

委托的四个层级：

| 层级 | 机制 | 你做什么 | AI 做什么 | 委托程度 |
|------|------|---------|----------|----------|
| **L1 手动** | 直接命令 | 检查、决策、下指令 | 执行 | 0% |
| **L2 Cron** | 定时任务 | 定精确时间"9点做" | 到点执行 | 30% |
| **L3 Heartbeat** | 周期检查 | 定周期"约30分钟检查" | 自己判断该不该说 | 60% |
| **L4 Hooks** | 事件响应 | 定规则"发生时做" | 自动反应，无需判断 | 90% |

---

## 2. 每个层级的关键洞察

### L2 Cron：工业时间思维
- 精确、机械、可预测
- 局限：不会自己判断"今天有没有必要做"
- 适合：必须精确时间的任务（日报、定时备份）

### L3 Heartbeat：有机时间思维
- 灵活、自主、降噪
- 核心：`HEARTBEAT_OK` = "没事，别打扰你"
- 适合：批量检查、监控类任务

### L4 Hooks：条件反射
- 事件触发、即时响应
- 例子：`/new` 命令 → 自动保存记忆
- 适合：生命周期事件、审计日志

---

## 3. 我的实践配置

### L1 手动（保留）
- 敏感操作：API key 修改、权限变更
- 创意决策：选题方向、文章风格调整

### L2 Cron（定时任务）
```bash
# 知识库自动处理
prism-auto-process: 每 60 分钟
prism-auto-output: 每 120 分钟

# 链接收集
link-collector: 每 30 分钟

# X.com 监控
@karpathy: 每 30 分钟
@a16z: 每 30 分钟
@openclaw: 每 60 分钟

# ClawHub 同步
blog-auto-sync: 每 2 小时
```

### L3 Heartbeat（周期检查）
```markdown
# HEARTBEAT.md

## 定时任务

### 1. 记忆维护
每次 heartbeat 时执行：
- 回顾当天对话，提取重要内容
- 更新 `memory/YYYY-MM-DD.md`
- 如有重大事项，同步更新 `MEMORY.md`
```

### L4 Hooks（事件响应）
- `session-memory`: `/new` 或 `/reset` 时自动保存记忆
- `command-logger`: 记录所有命令到审计日志
- `boot-md`: Gateway 启动时执行初始化

## 4. 委托的风险边界

什么时候该收回控制权？

| 场景 | 建议层级 | 原因 |
|------|---------|------|
| API key 操作 | L1 手动 | 安全敏感 |
| 权限变更 | L1 手动 | 需要审批 |
| 文章发布 | L2 Cron | 可定时，但需预览 |
| 知识库处理 | L3 Heartbeat | 批量检查，降噪 |
| 记忆保存 | L4 Hooks | 无副作用，自动即可 |

## 5. 心态转变：从操作员到监管者

最难的不是技术，是敢不敢放手。

**操作员心态**：
- "我得盯着它"
- "我怕它搞砸"
- "我每隔10分钟就想检查一下"

**监管者心态**：
- "我定好规则"
- "我抽查结果"
- "我只在异常时介入"
