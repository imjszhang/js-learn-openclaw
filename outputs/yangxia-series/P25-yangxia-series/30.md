# 你的知识系统，不用你维护了

> 系列：JS 的养虾日记 · 第 30 篇
>
> 关键词：知识管理、知识棱镜、自动整理、系统生长、复利效应

---

上周末，我翻出了半年前建的 Obsidian 知识库。

密密麻麻的分类、标签、双向链接。

文件夹按「技术 / 产品 / 生活 / 灵感」四层展开，每个文件夹里又有按月份分区的子目录。

标签系统用了三种：主题标签、状态标签、优先级标签。

双向链接建了几十个，还有几张 MOC（Map of Content）索引图。

**精致得像一座博物馆。**

然后我问了自己一个问题：

上一次打开它是什么时候？

想不起来了。

---

## 知识管理的三个死穴

我不是一个人遇到这个问题。

你去问问身边任何用过 Notion、Obsidian、语雀的人，十个里面有八个，知识库的现状跟我那个一样——建的时候热血沸腾，三个月后彻底忘记密码。

**死穴一：分类是人为的，知识是有机的**

你建分类的时候觉得完美。

"技术"下面分"前端 / 后端 / AI / 运维"，每个再分"入门 / 进阶 / 踩坑"。

但知识不按你的分类长。

你收藏了一篇"用 React 做 AI 前端"的文章——放前端还是放 AI？

一个月后，新内容跟旧分类对不上了。

要么放弃分类，要么放弃更新。

大多数人选前者。

**死穴二：维护成本 > 使用收益**

每加一条内容，你都要想三件事：

放在哪个分类？打什么标签？跟谁关联？

前三天你觉得有意思。

第七天你觉得烦。

第十四天你开始跳过这些步骤。

第二十天你不再打开它。

不维护的知识系统是什么？

**数字仓库。**

东西堆在里面，落灰，发霉，你再也找不到。

**死穴三：检索靠人，关联靠记忆**

你得知道搜什么关键词。

你得记得自己写过什么。

跨主题的关联？

几乎不可能手动建立。

因为你不可能同时记住技术笔记里的某个观点，跟三个月前收藏的那篇产品文章有什么关系。

**传统知识管理的根本问题就一个：**

**人是系统唯一的动力源。**

你停下来，系统就死了。

---

## 龙虾的知识系统：从仓库到生态

后来我换了个思路。

不再追求"建一个完美的分类系统"，而是想：**有没有一种知识系统，不需要我维护，它自己会变化？**

我让我的龙虾 JS_CLAW 帮我做这件事。

它做的事情，跟 Notion 完全不同。

**Notion 是仓库。** 你往里放东西，它就在那。你不去翻，它永远不动。

**龙虾的系统是生态。** 信息进来，它会自动拆解、归组、关联。你不需要维护，它在自己长。

怎么做的？

五个字：**收、拆、连、产、馈。**

[图片]

**收：丢链接，它自己来**

我看到一篇好文章。

不需要复制粘贴，不需要打开笔记软件，不需要想"放哪个文件夹"。

丢给龙虾一个链接。

它自动抓取网页内容，AI 总结概要、提取关键信息，存进知识库。

**拆：自动拆解，自然归类**

这是最关键的一步。

系统叫 js-knowledge-prism（知识棱镜），是我自己开发的一个工具。

它做的事情类似棱镜分光——把一篇完整的文章，拆解成一个个独立的信息单元。

js-knowledge-prism 管这些信息单元叫 **atoms**。

然后，它会自动把相似主题的 atoms 归到一组，叫 **groups**。

我不需要手动分类。分类是自然涌现的——内容相似的自然就在一起了。

**连：它自己会发现关联**

这是我自己做不到的事。

js-knowledge-prism 会分析不同 groups 之间的关联，生成更高层次的合成观点，叫 **synthesis**。

某天 JS_CLAW 突然跟我说：

> "你上个月收藏的那篇关于 Claude Code 源码的文章，跟你最近在写的 Harness Engineering 那篇有关联。要不要看看？"

我自己永远想不到这两件事有关系。

但系统看到了：Claude Code 源码里提到的配置结构，跟 Harness Engineering 的核心概念高度重叠。

这就是「系统自己长出来」的体现。

**产：素材已经长好了**

写 KL27（就是上一篇微信数据出口那篇）的时候，我体会到了这种变化。

如果是以前：
1. 打开 Notion 搜关键词，翻几篇笔记
2. 打开 Obsidian 再搜一遍
3. 打开微信收藏一篇篇翻
4. 自己整理大纲，从零开始写

**那天我是这么做的：**

1. 告诉 JS_CLAW：「帮我整理微信数据出口相关的素材」
2. 它自动去知识库里找相关的 atoms 和 groups
3. js-knowledge-prism 基于这些素材生成了文章框架
4. 我在框架基础上修改、润色、加入自己的观点

从 3 小时 → 1 小时。

不是因为我写得更快了。

是因为我不需要从零开始了。

素材已经在那里了，从 60 分开始写。

**馈：读者的反馈变成新素材**

读者的留言、提问、讨论，我都记录下来。

变成新的 journal（日记），丢进知识库。

系统继续消化、归组、关联。

**完整的闭环跑起来了：**

收 → 拆 → 连 → 产 → 馈 → 收……

你往里丢东西，系统自己长。

不需要你维护。

---

## 诚实的账本

说到这里，我想算一笔诚实的账。

**投入了多少？**

截至今天，我的知识库里：
- 文章入库 1359 篇（微信 957 + GitHub 195 + 知乎 170 + 其他）
- 信息单元 200 多个 atoms
- 知识分组 60 个 groups
- 合成观点 30 多个 synthesis

外加开发和维护 collector、prism 两个工具的时间。

**产出了什么？**

我不想说"一天能写 3 篇文章"——没有。

也不想说"篇篇爆文"——也没有。

诚实的说法是：

- 写新文章的时间从数天缩短到一天内
- 每篇文章都有素材支撑，不是凭空想象
- 系统自动发现的关联，经常超出我的预期

**复利体现在哪里？**

不在产出量上。

体现在「每次输入，让下一次输出更轻松」。

写第 1 篇文章的时候，我花了 3 天，从空白文档开始。

写第 27 篇的时候，我花了 1 天，素材和框架都是系统准备好的。

效率提升了三分之二。

但更重要的不是效率。

是**质量没有下降**。

有素材支撑的文章，跟从零开始硬憋的文章，读起来是完全不同的。

前者有细节、有数据、有具体的场景。

后者是空壳，是"在数字化浪潮中"那种套话堆砌。

**我还没看到"复利爆发"的那一天。**

系统还在长。

但我每天都在往里投入，它每天都在变强。

方向是对的。

这就够了。

## 跟 Karpathy 不谋而合

后来我看到 Karpathy 分享他的知识管理方法。

他用 Obsidian，但用法跟我以前完全不同。

他不建复杂的分类系统。

他只建一个简单的 inbox（收件箱），把所有看到的想法、笔记、文章摘要丢进去。

然后让笔记之间自然生长关联。

**他的核心观点跟我一样：**

不要预设分类。让分类从内容中长出来。

不同的是，他是手动做的。

我是让龙虾自动做的。

但底层逻辑是同一个：

**知识的价值不在单条信息，在信息之间的关联。**

你往 Notion 里存 1000 篇文章，那不叫知识管理，叫数字仓库。

你把 1000 篇文章拆成 5000 个信息单元，让它们自动关联成 100 个主题，那叫知识生态。

[图片]

仓库是死的。生态是活的。

## 你可以做什么

你不需要装 Notion，不需要建 Obsidian，不需要花 100 小时搭一个精美的分类系统。

你可以做的最小的第一步：

**找一个能自动收集信息的工具。**

让它帮你总结、入库。

然后定期回顾一次——每周或者每月。

就这些。

**核心原则四条：**

**低维护成本。** 系统自己能做的事情，不要让人做。

**自然涌现。** 不要预设分类，让分类从内容中长出来。

**关联优先。** 单条信息的价值有限，信息之间的关联才有价值。

**长期投入。** 知识系统是种树，不是种菜。种菜当天就能吃，种树要等三年。

**诚实的预期管理：**

第一个月，你可能觉得「没什么变化」。

第三个月，你会开始发现「好像有些东西在自动关联」。

第六个月，你会意识到「系统比我记得还清楚」。

一年之后，你会有「种了一棵树，它自己长成了森林」的感觉。

---

## 种一棵树

最好的知识管理系统，是你不需要打开它，它会来找你。

知识管理不是整理，是种植。

整理是消耗，种植是生长。

整理是把东西摆整齐，种植是让东西自己长出来。

与其花 100 小时整理一个精美的数字仓库，不如花 1 小时种一棵会自己长的树。

然后等它长大。

这就是我跟我的龙虾一起做的事。

### 知识系统核心流程
```
收（丢链接） → 拆（提取 atoms） → 连（生成 groups + synthesis） → 产（素材框架） → 馈（记录新想法）
```

### 核心概念
| 概念 | 说明 |
|------|------|
| atom | 最小信息单元——从文章中提取的独立知识点 |
| group | 相似 atoms 的自然归组——类似"自动生成的标签" |
| synthesis | 跨 groups 的高层合成观点——系统自己发现的关联 |
| journal | 日记模式——每天一个目录，记录真实工作和思考 |
