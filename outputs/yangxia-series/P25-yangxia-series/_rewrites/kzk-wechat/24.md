## 一、死掉的 Harness:配置思维的陷阱

三个月前,我干了件蠢事。

我花了一周时间,给 JS_BestAgent 设计了一套"完美"的系统指令。

500 行 Markdown。

密密麻麻。

涵盖了 20 种内容分类规则,50 条硬性约束,还有详细的命名规范和输出模板。

我当时觉得自己牛逼坏了。

这套配置简直就是 Harness 工程的教科书。

**三天后,现实给了我一耳光。**

Karpathy 发了一条关于 LLM Knowledge Base 的推文。

我需要 JS_BestAgent 分析这条推文,并关联到我"养虾日记"的历史内容。

坏事了。

我的"完美"配置里,根本没有"跨文章关联分析"这个能力。

我傻眼了。

摆在我面前的只有两条路。

要么,手动写一段 Prompt 临时救场。

要么,在那 500 行代码里像大海捞针一样找到位置插入新规则,还得祈祷别把原有逻辑搞崩。

选 1?

那意味着我那一周的"完美"设计就是个笑话,关键时刻还得靠人工补丁。

选 2?

那比重写还痛苦,简直是想死。

**这就是配置思维的陷阱:你试图用静态的设计去捕捉动态的需求。**

人类又不是神,哪能预见所有场景?

你花一周设计的"完美"系统,三天后就会遇到第一个它处理不了的坑。

然后第二个,第三个。

到最后,你宁愿让需求将就现有的配置,也不愿推翻重来。

系统成了你的负担,而不是助力。

## 二、Karpathy 的启示:聊出来的系统架构

上个月,Karpathy 分享了他的 LLM Knowledge Base 工作流。

35K 赞,6.3 万收藏。

评论区那帮人还在问"用的什么工具"。

真看懂的人都知道--这不是工具推荐,是**范式预告**。

他说了一句很关键的话:

> *"You rarely ever write or edit the wiki manually, it's the domain of the LLM."*

翻译成人话就是:你几乎不需要手动编写或编辑 wiki,那是 LLM 的地盘。

但最让我震惊的不是这句话。

是他**没做什么**。

他没有设计复杂的 RAG 架构。

他没有预设详细的分类体系。

他没有写几百行的系统指令。

他只是把原始素材丢进 `raw/` 目录。

然后开始和 LLM 对话。

就这么简单?

就这么简单。

**结果,系统自己"长"出来了:**

自动维护的索引文件。

动态生成的概念分类。

自然形成的反向链接。

数据规模多大?

~100 篇文章,~40 万词。

他以为得用那些 fancy 的 RAG 技术。

实际测试发现--**不需要**。

因为 LLM 自动维护了索引和摘要,在这个规模下能轻松读取所有重要相关数据。

**关键洞察:活的系统架构是聊出来的,不是设计出来的。**

Karpathy 没有预测未来会遇到什么查询场景。

他只是开始对话。

让系统在对话中自我组织、自我演化。

这不是偷懒。

这是一种新的开发范式。

## 三、Harness Engineering 的深层推演

今年 2 月,OpenAI 发了一篇文章:*Harness Engineering: Leveraging Codex in an Agent-First World*。

他们的工程师用五个月生成了大约一百万行代码。

注意,没有一行是人手写的。

文章强调了 Harness 的重要性。

系统指令、工具配置、记忆管理,这些决定了 Agent 的上限。

但文章里有个坑,它没明说。

**好的 Harness 不是一次配置完美的,而是在持续交互中自我迭代的。**

看看我的养虾系统是怎么演化的:

- **第 1 个月**:JS_BestAgent 只有简单的系统提示词,像个婴儿。
- **第 2 个月**:通过 200+ 次对话,沉淀出任务分发规则,开始懂事。
- **第 3 个月**:通过 500+ 次对话,演化出子 Agent 协作机制,学会 teamwork。
- **第 4 个月**:通过 800+ 次对话,形成完整的 Harness 体系,彻底成年。

现在的养虾系统有 4 个子 Agent 在并行干活:情报组、资料组、知识组、汇总组。

这个架构不是我在某天坐下来,抽着烟设计出来的。

而是在 800+ 次对话中**长**出来的。

每一次对话都在改进 Harness。

不是我去改配置。

而是系统通过对话自我调整。

**配置是预测未来,对话是创造未来。**

你品品,这味道对吗?

## 四、聊天即开发范式

这让我想到编程范式的演进。

**命令式编程**(C、汇编):告诉计算机每一步怎么做,累得像条狗。

**声明式编程**(SQL、React):告诉计算机想要什么结果,稍微轻松点。

**对话式编程**(Chat-as-Development-Paradigm):通过对话让系统自我演化,爽翻天。

前两种范式,人类都是"设计者"。

要么设计执行步骤,要么设计目标状态。

对话式范式,人类是"对话者"。

你不设计系统。

你和系统对话,让它自己长出来。

这不是降级。

是升级。

因为**系统的复杂度超过了人类的预见能力**。

你无法在设计阶段就考虑所有边界情况。

唯一的办法,是让系统在使用中自我适应。

Karpathy 的 wiki 是这样长出来的。

我的养虾系统是这样长出来的。

未来最好的 Agent,都会是这样长出来的。

## 五、回到开头

我说过一句得罪人的话:从 Harness 角度来看,现在的所有 Agent 都是垃圾。

现在我要收回一半。

不是所有 Agent 都是垃圾。

只有那些"死"的 Harness 才是垃圾。

预设好就固定不变,无法自我进化,那就是电子垃圾。

如果你的 Agent 能在对话中迭代、在使用中成长。

它不是垃圾。

它是活的。

**配置是死的,对话是活的。**

最好的系统不是设计出来的,是长出来的。

而最好的 Agent,一定是聊出来的。

这是 Harness Engineering 的终局:**聊天即开发范式**。
