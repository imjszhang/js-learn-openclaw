**它的强大不靠模型，靠工程。**

这是 Raschka 拆解完 Claude Code 后，最狠的一句结论。

六大技术杀手锏，没一个是改模型权重的。

全是工程活。

实时加载仓库上下文。

激进地复用 Prompt 缓存。

专用工具链代替裸 Bash。

把膨胀的上下文压缩到吐。

会话记忆结构化。

Fork 子 Agent 并行跑。

翻译成大白话就是：

**别让模型干脏活，把上下文管好比什么都强。**

我刚开始装 OpenClaw 时，也犯过这傻逼错误。

以为 Prompts 写得越长，龙虾就越聪明。

结果塞进去几万字规则。

它反应慢得像便秘。

说着说着，连自己是谁都忘了。

后来我搞出了分层注入。

`SOUL.md` 定人格。

`AGENTS.md` 定规矩。

`USER.md` 定偏好。

`IDENTITY.md` 定名片。

每次启动，只加载最核心的。

其他的，让它按需去查。

这和 Claude Code 的 `CLAUDE.md` 是一个路子。

**把静态上下文固化到文件系统，让动态上下文只承载当前任务。**

社区提炼出的 8 个设计模式，几乎覆盖了“如何构建靠谱 AI Agent"的所有核心问题。

我来一条条对照，看看 OpenClaw 是怎么玩的。

**1. 协调者模式**

核心就一条：禁止“懒委托”。

别写“基于你的发现，修复这个 bug"。

这种话就是放屁。

必须消化研究结果，给出精确到文件路径和行号的指令。

我在 KL11 里写过，让龙虾写代码时，我从不扔一句“修一下”。

我把上下文、预期结果、验收标准全甩给它。

让它出方案，我审核，再执行。

OpenClaw 怎么做？

用 `sessions_spawn` 启动 ACP 子代理。

主龙虾只做决策。

子代理去干具体的脏活。

**2. 任务并发模式**

只读任务？随便并行。

写操作？同一文件区域必须串行。

用 `AsyncLocalStorage` 做隔离。

我的 `js-knowledge-collector` 就是这个逻辑。

抓取文章是只读，可以并发跑。

写入数据库是写操作，必须排队。

用 inbox/batch 轮转，确保队列不乱。

OpenClaw 实践：`cron` 定时任务隔离 + `sessions_spawn` 独立会话。

**3. 对抗性验证**

目标不是确认实现正确。

是尝试打破它。

我在 KL12 里写过 ACP 代码审查。

Cursor Agent 生成的代码，我绝不直接采纳。

我会让另一个实例专门找茬。

有两个已知失败模式：

读了代码就写 PASS，从不跑命令。

看到测试通过就放行，没注意一半功能是空的。

OpenClaw 实践：多 Agent 交叉验证 + 自动化测试。

**4. 自我合理化防护**

这条最狠。

也是我在养虾过程中摔过最大的坑。

识别并阻止 AI 的自我合理化。

当 AI 说“代码看起来正确”时。

正确行动是：运行它。

以前我让龙虾改配置，它说“应该没问题了”。

我信了。

结果服务起不来，我想死的心都有。

现在我强制要求：

**任何配置变更，必须先备份，再应用，再验证，三步缺一不可。**

OpenClaw 实践：`HEARTBEAT.md` 自检任务 + `memory_search` 事实核查。

**5. Worker 指令编写**

Worker 看不到你的对话上下文。

每条指令必须自包含。

文件路径、行号、完成标准，一个都不能少。

这和 `SKILL.md` 的设计完全一致。

我的每个技能都有标准化的输入/输出定义。

Worker（子代理）不知道主会话的历史。

只按给定的上下文执行任务。

OpenClaw 实践：`SKILL.md` 模板标准化 + `tools` 定义明确输入输出。

**6. 记忆类型系统**

记忆分四类。

User（画像）、Feedback（纠正）、Project（状态）、Reference（指针）。

同时列出“绝对不记”清单。

这就是我在 KL13 里写的“三层记忆系统”细化版。

`MEMORY.md` 对应 User。

`memory/YYYY-MM-DD.md` 对应 Feedback。

`knowledge_prism/Groups` 对应 Project。

`knowledge_prism/Atoms` 对应 Reference。

OpenClaw 实践：三层记忆架构 + Prism 分层处理。

**7. 记忆防护**

三道防线。

漂移防护：行动前验证文件还在不在。

膨胀检查：超 5KB 自动瘦身。

写入过滤：6 个月后还有用吗？没用就滚。

我配置 `memory_search` 时，用了时间衰减（半衰期 30 天）+ MMR 重排序。

本质上就是在做记忆膨胀检查。

太旧、相关性太低的记忆，自动降权。

OpenClaw 实践：`memorySearch` 配置中的时间衰减 + 嵌入缓存限制。

**8. 轻量探索**

探索任务三个属性：只读、快速、低成本。

不知道位置？广搜。

知道位置？精确读。

搜不到？换策略。

独立搜索必须并行。

我的 `js-eyes` 浏览器控制就是这个逻辑。

需要看网页？直接调用 `js_eyes_get_html`，轻量只读。

需要深度抓取？再用 `knowledge_collect` 全链路处理。

OpenClaw 实践：`web_search`（广搜）+ `js_eyes_get_html`（精读）+ `knowledge_search`（向量检索）三源互补。

**Buddy System**：内置电子宠物系统。

**KAIROS**：7x24 小时自主运行架构。

Karpathy 点评：这些功能明显是将 Claude Code“龙虾化”。

我笑了。

这不就是我在 KL09 里写的 Harness Engineering 四层架构吗？

L1 被动响应 → 你问我答。

L2 主动提示 → `HEARTBEAT.md` 定期检查。

L3 任务委托 → ACP 外包给 Cursor。

L4 自主运行 → `cron` 定时任务 + KAIROS（如果我写的话）。

Buddy System 对应 `nodes` 设备配对通知。

KAIROS 对应 `cron` + `HEARTBEAT.md` 的自治架构。

结论很简单：

**顶级产品的终极目标，和我养虾的方向完全一致——让 AI 从「工具」变成「员工」，再变成「自主代理」。**

很多人把这当八卦看。

我看到的是一次“工程透明化”实验。

51 万行代码裸奔，证明了一件事：

**工程架构才是真正的护城河。**

模型可以换。

开源模型塞进同样框架也能很强。

但实时上下文加载、激进缓存、专用工具链、结构化记忆。

这些工程优化的总和，才是用户体验的决定性因素。

而 OpenClaw 的分布式 Skill 架构，比 Claude Code 的巨型单体系统更优。

你看这张表。

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| 架构 | 51 万行单体 | 分布式 Skill |
| 可定制 | 闭源，只能调 API | 开源，可自行开发 Skill |
| 成本 | 按 Token 计费 | 本地部署 + ACP 外包，省 90% |
| 数据 | 上传云端 | 完全本地，零泄露风险 |
| 生态 | Anthropic 一家 | Moltbook 等多平台 |

**泄露事件反而证明：分散的 Skill 生态 > 巨型单体系统。**

就算 OpenClaw 的某个 Skill 源码泄露（本来就是开源的）。

也不会像 Claude Code 这样“底裤被扒光”。

因为每个 Skill 职责单一。

替换成本低。

整个系统的韧性远高于单体架构。

---

如果你也想把 OpenClaw 配置得像 Claude Code 一样强。

别瞎琢磨了，按这个清单来：

1. **分层上下文**：`SOUL.md` + `AGENTS.md` + `USER.md` + `IDENTITY.md`
2. **记忆三层**：`MEMORY.md`（长期）+ `memory/*.md`（短期）+ `knowledge_prism`（结构化）
3. **并发隔离**：`sessions_spawn` ACP 子代理 + `cron` 定时任务
4. **对抗验证**：多 Agent 交叉检查 + 自动化测试
5. **工具链专用**：不写裸 Bash，用专用 Skill（`js-eyes`、`knowledge_*`）
6. **自我防护**：`HEARTBEAT.md` 自检 + `memory_search` 事实核查
7. **成本控制**：重活外包给 Cursor（ACP），主龙虾只做决策

这七条。

和 Claude Code 的六大技术、八大 Skill 一一对应。
