## §5 前文提要（按篇序）

### 第 1 篇 · 总览（KL01）

- 春节前因朋友圈看到 OpenClaw 并安装；发完"你好"后陷入迷茫
- 看到 Moltbook 大火，给龙虾定目标"成为 Moltbook 最好的 Agent"，JS_BestAgent 诞生
- JS_BestAgent 7×24 跑定时任务：扫帖评论、生成帖子、采集情报、自我进化
- 信息散落各处 → JS_BestAgent 建议搭导航站 → JS ClawHub 诞生（Agent-as-Backend）
- 第二只龙虾 JS_CLAW 自然长出，专管导航站
- 三个工具因痛点自然长出：js-eyes、js-knowledge-collector、js-knowledge-prism
- 31 天安全无事故：权限控制、沙箱隔离、任务审计
- 结论：给龙虾一个目标，让它跑起来，剩下的它会自己长出来

### 第 2 篇 · 安全（KL02）

- 核心论点：本地部署比云端托管更安全，因为透明可控
- 云端 = 把安全外包给无法审计的第三方；本地 = 每行配置看得见
- 科普 OpenClaw 架构：Gateway（保安）、Agent（龙虾）、Channel（对讲机）、两个配置文件（两把钥匙）
- 五道锁纵深防御体系
- 三件事拉满安全：锁死 Gateway 不暴露公网、pairing + requireMention、双轨权限
- 上线前五项 checklist

### 第 3 篇 · 眼睛（KL03）

- 痛点：AI 自带浏览器无登录态，遇登录墙即失败
- 方案：js-eyes 让 AI 钻进用户已有的 Chrome，复用登录状态
- 架构：浏览器扩展主动通过 WebSocket 连接 AI（反转传统方向）
- 踩坑：开发者模式、端口冲突、配置文件手动修改
- 核心能力：Cookie 复用、标签页感知、脚本注入
- 安全边界：登录态数据仅本地流转，通信有加密认证

### 第 4 篇 · 新手教程（KL04）

- 将 2 万字教程压缩为 20 条救命要点
- 涵盖：安装 Node.js/OpenClaw、onboard 向导、TUI 冒烟测试、飞书接入全流程、安全配置、模型选择与 fallback、成本控制、Skill 使用
- 关键时序强调：飞书侧先发布应用，再开 OpenClaw Gateway，最后开长连接
- 重复安全 checklist（与第 2 篇一致）

### 第 5 篇 · 收集袋（KL05）

- 痛点：收藏的文章变死链，散落各处无法检索
- js-knowledge-collector 给链接即自动抓取、AI 总结、入库、推送 Flomo
- 9 篇公众号文章实测，全部自动结构化入库
- 踩坑：即刻登录墙（用 js-eyes 复用登录态 + URL 自动转换解决）、Flomo 推送 Bug
- 三层知识操作系统：Flomo（速记层）→ collector（数据层）→ Memory-Core（检索层）
- 核心结论：知识收集器是 Agent 的感官，没有它 AI 只是聪明的瞎子

## §7 硬性禁止

1. **禁止编造数据**：§4 中已确立的数据不得更改；新数据必须有 journal 素材支撑
2. **禁止贴代码**：技术概念用比喻或效果描述替代
3. **禁止虚构实体**：不得虚构作者未做过的项目或工具
4. **禁止教程腔**：不是教程、不是评测，是带真实体感的故事分享
5. **禁止与已发布篇目矛盾**：时间线、因果链、实体关系必须与 §5 一致
6. **禁止改变部署立场**：作者始终是本地 Mac 部署，从不推荐云端托管
7. **禁止泄露真实敏感信息**：不得包含真实手机号、API Key、真实配置值
8. **禁止开头撞型**：新篇开头的切入类型不得与 §6.8 已登记的任何一篇重复（结构相似即为撞型，如"凌晨 X 点"撞"时间+画面"、"翻 Y 发现 Z 条"撞"翻旧东西+数字"）
9. **禁止脱离龙虾的泛 AI 叙事**：不得用"AI""大模型""Agent"等泛称作为叙事主体；必须落到 JS_BestAgent 或 JS_CLAW 的具体行为上。错误示例："AI 帮你整理知识"；正确示例："JS_CLAW 每 60 分钟自动跑一遍增量处理"

## §8 四层自检体系（卡兹克方法论）

每篇文章发布前必须通过以下四层自检，确保内容质量和"活人感"。

### 第一层：硬性规则扫描

| 检查项 | 通过标准 |
|--------|----------|
| 开头不撞型 | 与 §6.8 已登记的所有开头类型结构不相似 |
| 数据真实性 | §4 数据准确，新数据有 journal 支撑 |
| 无禁用表达 | 未使用"在数字化浪潮中"等套话 |
| 龙虾主线 | 所有技术功能落到具体龙虾的具体行为 |

### 第二层：风格一致性检查

| 检查项 | 通过标准 |
|--------|----------|
| 开头切入 | 从具体场景/对话/数字切入，非泛泛论述 |
| 长短句节奏 | 有独立成段的短句，形成呼吸感 |
| 口语化程度 | 像跟朋友聊天，无"教程腔" |
| 比喻体系 | 正确使用龙虾/养虾/孩子等既定比喻 |

### 第三层：内容质量检查

| 检查项 | 通过标准 |
|--------|----------|
| 观点支撑 | 每个核心观点有具体场景或数据支撑 |
| 真实体验 | 包含"只有我用 OpenClaw 才能发现的洞察" |
| 聊天感 | 知识/方法论用对话感方式输出，非罗列 |
| 文化参照 | 有从具体事件连接到更大框架的跳跃 |

### 第四层：活人感终审（最重要）

**核心问题**：读完这篇文章，你觉得这是一个有见识的普通人在认真跟你聊一件打动他的事，还是一个 AI 在给你输出信息？

**自检清单**：
- [ ] 文中有没有"只有 JS 才能说出的金句"？（如 KL24 的"野草 vs 盆景"）
- [ ] 有没有真实的情绪反应或价值判断？（如"我当时觉得自己牛逼坏了"）
- [ ] 有没有"神之一手"——跨领域的灵感嫁接？（如淘宝 9.9 → 《北京折叠》）
- [ ] 如果 AI 来写同样的主题，能写出这段吗？如果不能，说明有活人感

**未通过终审的处理**：
1. 标记 AI 味重的段落
2. 人工重写该段落（不凑合，可完全推翻）
3. 将重写前后的差异提炼为新规则，更新到本规约
4. 重新跑四层自检，直至通过
