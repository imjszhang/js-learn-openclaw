# 教你的龙虾，用你的 flomo：从被动记录到主动思考的进化

> Day 77 · 2026-04-17 · 骨架 [KL49](./P23-practice-diary/KL49-2026-04-17.md)

今天终于跨过了“系统自生长”到“系统替你思考”的关键门槛。面对 flomo 里堆积如山的笔记，我意识到单纯解决“数据获取”（如 KL27 的 wechat-cli）已不够，真正的痛点是“记了但读不完”，必须让系统具备自动消化和洞察的能力。

## 直击痛点：从“龙虾够不到”到“龙虾读不完”

在构建 KL30 实现知识库自动维护后，我发现了一个更深层的困境：flomo 用户的核心问题不再是“没记”，而是“记太多读不过来”。搜索一个关键词跳出 200 条结果却只能关掉窗口，或者明明写过却想不起具体内容，这种人脑无法同时处理海量信息的瓶颈，迫使我将系统定位从“辅助记录”转向“替代阅读”。js-knowledge-flomo 项目的诞生，正是为了解决“龙虾读不完笔记”的问题，让系统先读，再告诉用户它发现了什么，实现从数据积累到认知升维的跨越。

## 架构支撑：Node.js+SQLite+LLM 的五层模块化设计

为了支撑这一目标，我采用了轻量级但高扩展的技术栈，构建了包含 CLI、MCP Server 及 Web UI 的多端协同架构。项目基于 Node.js 18+，利用 SQLite 作为零依赖的本地缓存，通过 OpenAI 兼容 API 调用 LLM，并严格遵循 flomo 官方 MCP 协议。核心目录结构清晰划分：`cli/` 处理 auth、search 及 insights 等核心逻辑，`mcp-server/` 提供 IDE 集成能力，`prompts/` 独立管理提示词模板以实现与代码解耦，而 `openclaw-plugin/` 则让龙虾能直接调用 15 个 flomo 工具。这种 MCP Client + MCP Server 的双角色设计，确保了用户能在终端、IDE 或本地网页中灵活接入自动化工作流。

## 核心能力：七大 AI 分析功能实现主动思考

系统的“主动思考”能力具体落地为七大核心分析功能，深度应用 LLM 进行跨时空的隐藏关联挖掘。首先是**洞察发现**，系统自动按标签搜索笔记缓存至 SQLite，分析反复主题、情绪变化及未被注意的关联，回答“你最近真正在关心什么”；其次是**关联发现**，强制 LLM 跨标签寻找至少 3 组有意义的连接，推测用户思考的大问题；**观点演变**功能则按时间线标注关键转折点，清晰展示立场从 A 到 B 的证据。此外，系统还能基于真实散落笔记生成**写作大纲**，通过“相关推荐”接口进行**素材搜集**，利用现有标签树为无标签笔记**推荐标签**，甚至对标签体系进行**审计**，指出重复、命名不一致或层级问题并给出合并建议。这些功能不再依赖人工整理，而是由系统自动完成从素材到结构的转化。

## 安全底座：OAuth 授权与本地缓存的隐私防线

在追求自动化的同时，我严格执行了安全隐私原则，确保数据不出用户环境。接入层选用 flomo 官方 MCP 协议，利用 OAuth 授权避免逆向工程带来的安全风险；数据存储层坚持使用 SQLite 本地缓存，轻量且支持 SQL 查询，杜绝云端泄露隐患；AI 分析层强制走用户自建的 LLM API，确保原始笔记内容不经过第三方模型服务商。同时，我将所有 Prompt 模板独立存放在 `prompts/` 目录下，不仅便于单独优化迭代，更实现了提示词工程与业务逻辑的彻底解耦，为系统的长期可维护性打下坚实基础。

## 今天的收获

- **痛点演进识别**：知识管理的瓶颈已从“数据获取难”（KL27）转移为“信息消化难”（KL31），系统定位需从“工具”升级为“代理”。
- **架构解耦策略**：将 Prompt 模板独立目录管理，实现了 AI 逻辑与代码逻辑的解耦，便于非开发人员优化提示词而不触碰代码。
- **安全隐私红线**：坚持"OAuth 授权 + 本地 SQLite 缓存 + 自建 LLM API"三位一体，是构建可信个人知识系统的必要底线。
- **多端协同形态**：通过 MCP Client/Server 双角色设计，一套代码即可覆盖 CLI、IDE 插件、OpenClaw 技能及 Web UI 四种场景，最大化复用率。
- **主动思考落地**：真正的智能不是存储更多，而是能自动输出“观点演变”、“隐藏关联”和“写作大纲”，将被动记录转化为主动认知。
