# 从漏看到闭环：构建 X.com 账号自动化监控的实战路径

> Day 60 · 2026-03-31 · 骨架 [KL38](./P23-practice-diary/KL38-2026-03-31.md)

今天的主线任务非常明确：解决因时区错位和信息过载导致的“漏看”痛点，利用 `js-eyes` 扩展构建一套针对 @karpathy、@sama 和 @paulg 三个高价值账号的自动化监控闭环。这不仅是养虾日记 014 的选题落地，更是对 OpenClaw 网关在“带登录态监控”场景下的一次关键压力测试。

## 基于 JS-Eyes 登录态复用的低侵入式抓取架构

面对 X.com 严格的反爬机制，我决定放弃传统的 API 调用或无头浏览器方案，转而采用“本地优先”的架构策略。核心思路是利用 `js-eyes` 浏览器扩展直接复用我当前浏览器中已存在的登录态（Cookie/Session），彻底绕过身份验证难题。

具体实现上，我设计了如下数据流转链路：`X.com 时间线` → `JS-Eyes 扩展` → `WebSocket` → `OpenClaw Gateway` → `JS_BestAgent/Cron` → `飞书消息推送`。在 Agent 侧，我调用了 `js_eyes_open_url` 打开目标主页获取 `tabId`，随即通过 `js_eyes_execute_script` 注入脚本提取帖子数组。这种架构不仅实现了“只读不写”的安全边界（不点赞、不转发），更将抓取逻辑完全下沉到客户端，极大降低了服务端的封号风险。

## 实施"ID 优先 + 内容哈希”的双重去重策略

在数据落地环节，单纯的帖子 ID 比对不足以应对 X.com 用户编辑推文后 ID 不变但内容变化的场景。因此，我确立了“方案 B：内容哈希去重”策略。系统会在 `~/.openclaw/workspace/x-monitor/state.json` 中持久化存储每个账号的状态，数据结构包含 `lastCheck` 时间戳以及 `posts` 数组。

每个帖子对象记录了 `id`、`hash`（SHA256 内容哈希）、`timestamp` 和 `notified` 标志。比对逻辑被设计为双重校验：先查询帖子 ID 是否存在，若存在则进一步比对内容哈希。只有当 ID 不存在或哈希值发生变化时，系统才会判定为新消息并触发后续的 LLM 摘要与飞书推送。这一机制确保了即使推文被修改，我们也能精准捕获最新的高价值信息，同时避免重复通知的骚扰。

## 建立应对动态加载与 Rate Limit 的纵深防御

X.com 作为 React 单页应用，其动态加载特性是自动化抓取的最大拦路虎。在编写解析脚本时，我发现首屏仅显示 10-20 条内容，必须模拟滚动才能加载更多。为此，我在 `execute_script` 中加入了模拟滚动逻辑，并优先使用 `article[data-testid="tweet"]` 这类稳定的 `data-testid` 属性作为 DOM 选择器，而非易变的 class 名，同时准备了备用选择器以应对未来的页面重构。

针对 Rate Limit 和登录态失效风险，我配置了三个独立的 Cron 任务（ID 分别为 `40d76c72...`、`ae46e2a0...`、`c79f964b...`），将执行频率控制在每小时一次，并设置了 120 秒的超时限制。策略上，我引入了随机延迟来模糊请求特征，并在 Agent 逻辑中增加了登录态检测：一旦检测到 Cookie 失效导致抓取失败，立即发送通知提醒人工介入重新登录，从而保障系统的长期稳定性。

## 验证低成本摘要模型与演进路径

在完成框架搭建后，我对运行成本进行了详细测算。预计每月页面抓取消耗约 1M tokens，内容摘要（假设每天 10 条新帖）消耗约 300K tokens，合计 1.3M tokens。基于当前 Kimi K2 模型的价格，月均成本仅为 0.3-0.5 美元，这在可接受范围内。

接下来的七天（03-31 至 04-05）将是关键的验证期。今天的任务是完成框架搭建与首次测试，明日将跑通全流程并记录首次通知。我将重点收集通知总数、高价值帖子占比、踩坑次数以及实际 Token 消耗与预估值的偏差。这些数据将直接支撑养虾日记 014 的实战章节，验证从“技术框架”到“生产运行”的闭环可行性。

## 今天的收获

- **登录态复用是突破反爬的关键**：利用浏览器扩展复用本地 Session 比维护复杂的代理池更轻量、更安全，特别适合个人化监控场景。
- **双重去重机制保障信噪比**：结合“帖子 ID"与“内容哈希”的比对策略，能有效识别推文编辑行为，避免遗漏重要更新或产生重复通知。
- **防御性编程需前置**：针对 React 动态加载和 DOM 频繁变动，必须在脚本阶段就引入模拟滚动和稳定选择器（如 data-testid），而非事后修补。
- **低成本自动化可行**：通过合理的 Cron 调度（每小时一次）和轻量级摘要模型，可以将高价值信息监控的月成本控制在 0.5 美元以内。
