# 从 DOM 解析到封装 API：X.com 监控闭环的架构跃迁

> Day 60 · 2026-03-31 · 骨架 [KL38](./P23-practice-diary/KL38-2026-03-31.md)

今日主线任务是构建 X.com 账号自动化监控闭环，旨在解决信息过载与时区错位导致的“漏看”痛点。在从最初的 DOM 解析方案遭遇连续超时后，我果断切换至 `js-search-x` 封装 API 方案，不仅将单次执行时间从 70 秒压缩至 5-10 秒，更成功在 isolated session 中跑通了全链路，确立了低成本、高可用的监控基线。

## 构建基于 JS-Eyes 登录态复用与 WebSocket 转发的低侵入式抓取架构

上午 09:12，我首次尝试利用 `js_eyes_open_url` 打开目标主页并配合 `js_eyes_execute_script` 进行 DOM 解析。虽然 @karpathy 账号首次执行成功（耗时 70 秒），但第二次执行即告超时，而 @sama 和 @paulg 更是连续在 120 秒限制内失败。诊断发现，直接操作 React 动态渲染的 DOM 极不稳定，且容易触发 X.com 的反爬机制。

转折点出现在 09:45，我发现已安装的 `js-search-x` 扩展技能提供了 `x_get_profile` 接口。该接口直接复用浏览器扩展的登录态，通过 WebSocket 将提取的原始数据无缝流转至 OpenClaw 网关，完全绕过了传统 API 的反爬限制。我立即决定全面切换技术栈：废弃不稳定的 DOM 解析脚本，转而调用封装好的 `x_get_profile(username)` 工具。这一架构调整不仅解决了信息获取的稳定性问题，更展示了 JS-Eyes 在“带登录态监控”场景下的独特优势。

## 实施"ID 优先 + 内容哈希”的双重去重策略与结构化状态持久化

为确保系统在重启后仍能精准识别新增高价值信息，我设计了严格的状态持久化机制。去重逻辑采用“方案 B"：首先正则提取帖子链接中的 ID（匹配 `/status/([0-9]+)`），若 ID 存在则进一步比对内容哈希（SHA256），有效防止因帖子编辑产生的重复通知。

状态数据被结构化存储于 `~/.openclaw/workspace/x-monitor/{username}.json` 文件中。每个账号对象包含 `lastCheck` 时间戳及 `posts` 数组，数组内记录每条推文的 `id`、`hash`、`timestamp` 及 `notified` 状态。下午 10:45，在抓取过去 7 天历史数据建立基线时，我曾因子 Agent 混用工具导致 karpathy 账号数据遗漏（显示 0 条实际有 5 条）。通过强制指定使用 `x_get_profile` 工具并重新校验哈希，我成功修正了数据，最终建立了包含 37 条推文的完整历史基线，覆盖了从供应链攻击警告到产品更新的关键信息。

## 建立应对动态加载、Rate Limit 及登录态失效的纵深防御机制

针对 X.com 的动态加载特性与潜在的 Rate Limit 风险，我在 Cron 调度层面实施了多重防御。最初配置的每小时一次、超时 120 秒的任务在旧方案下频频失败。切换新方案后，我将超时阈值收紧至 60 秒，并将执行环境限定为 `isolated session`。

在 22:27 的关键验证中，我确认了 `js-search-x` 工具链可在 Cron 的 isolated session 中正常加载，成功获取了 @karpathy 关于 npm axios 供应链攻击的最新警告（8.4K 点赞）。为应对未来可能的登录态失效，架构中预留了检测机制：一旦抓取失败且判定为 Cookie 过期，系统将发送通知提醒人工重新登录，而非盲目重试导致封号风险。同时，通过控制频率（每小时一次）和随机延迟策略，我们在保障数据完整性的同时最大化降低了干扰。

## 验证低成本摘要模型可行性并确立从框架搭建到数据复盘的演进路径

基于 7 天历史基线数据（共 37 条推文，日均 5.3 条），我对系统成本进行了实测预估。`x_get_profile` 调用单次消耗约 200 tokens，配合针对高价值内容的摘要生成（约 800 tokens），预计 30 天总消耗仅为 777K tokens。按当前 Kimi K2 模型价格计算，月均成本约为 1.0 美元，远低于第三方监控工具（$20+/月）。

今日已完成框架搭建与历史数据清洗，标记出 5 条高价值案例，包括 karpathy 的 LLM 记忆洞察与 OpenClaw 的版本更新通知。接下来的演进路径清晰明确：从 4 月 1 日起进入 7 天实测期，重点观察 Cron 任务的首次自动触发、飞书通知的实际到达率以及信噪比表现，最终在 4 月 5 日完成从技术验证到生产运行的闭环报告。此外，我已将核心逻辑封装为开源 Skill `js-x-monitor-skill`，支持通过 `openclaw x-monitor add` 等命令快速扩展监控矩阵。

## 今天的收获

- **架构选型决定稳定性**：在动态渲染页面（如 X.com）的自动化场景中，优先选择复用登录态的封装 API（`x_get_profile`）而非原生 DOM 解析，可将执行成功率从 <50% 提升至 100%，耗时降低 90%。
- **双重去重保障信噪比**：单纯依赖帖子 ID 无法应对“编辑后重发”场景，必须结合“ID + 内容哈希”的双重校验机制，并将状态持久化至本地 JSON 以实现断点续传。
- **Isolated Session 的关键作用**：Cron 任务必须在 `isolated session` 中运行以避免上下文污染，但需提前验证扩展技能在该环境下的加载兼容性（如今日 22:27 的修复）。
- **低成本监控的可行性**：通过精准控制摘要触发条件（仅高价值内容）与低频轮询策略，可将大规模账号监控的月均 Token 成本控制在 1 美元以内。
- **历史基线的重要性**：在启动实时监控前，务必先抓取并清洗过去 7 天的历史数据建立基线，避免系统上线瞬间产生“通知风暴”。
