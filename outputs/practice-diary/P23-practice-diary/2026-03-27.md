# 把编码成本外包：ACP 架构让 OpenClaw 从「烧钱」转向「编排」

> Day 56 · 2026-03-27 · 骨架 [KL36](./KL36-2026-03-27.md)

今天终于捅破了 OpenClaw 高成本的那层窗户纸：编程任务不再由主对话模型硬抗，而是通过 ACP 协议完整外包给 Cursor CLI。这不仅是配置上的微调，更是从“可记录”迈向“可教学”的关键架构分层——让昂贵的 Token 只用于决策，让外部的 Harness 去承担繁重的读写与编码。

## 结构性降本：为何必须将文件读写剥离至外部运行时

过去几天复盘日志时，我发现 OpenClaw 的 Token 消耗大头并非日常对话，而是频繁的读文件、写代码和工具调用。每次这些操作都走 Chat 模型的上下文，成本累积极快。根本解决之道在于利用从 3.24 版本引入的 ACP（Agent Client Protocol）运行时。

ACP 允许 Gateway 将编程任务“外包”给外部 harness（如今天的 Cursor CLI），由外部工具自己的计费体系承担编码 Token。OpenClaw 退居二线，只负责编排和路由。我在 `acp-cursor-agent-integration.md` 中明确记录：这不是可选优化，而是从成本角度使用 OpenClaw 的结构性必要措施。通过将高消耗的 IO 操作物理隔离到 `js-cursor-agent` 插件驱动的外部进程中，我们实现了主对话与编码执行的彻底解耦。

## 配置激活：理清优先级与生效机制的陷阱

实施过程中第一个坑是配置未生效。虽然 `js-cursor-agent` 插件早已安装并实现了 `registerAcpRuntimeBackend({ id: "cursor" })`，但 Gateway 并未调用它。深入分析 `d:\.openclaw\openclaw.json` 后发现，`acp` 配置块完全缺失。

ACP 的配置遵循严格的覆盖顺序：`bindings[].acp` > `agents.list[].runtime.acp` > 全局 `acp.*`。必须在主配置中显式开启大门。我今天在 `gateway` 之前插入了 `acp` 块，设置 `enabled: true` 和 `backend: "cursor"`，并补全了插件的 `command`（指向 `agent.cmd`）、`model`（设为 `composer-1.5`）及 `permissionMode`。同时，参照 `acpx` 规范新建了 `openclaw.plugin.json`，声明了完整的 `configSchema` 和 `uiHints`，并果断移除了 `idleTtlMinutes` 等在 Gateway 模式下无效的限制字段，确保元数据纯净。

## 代码适配：硬编码覆写与独立 Session 隔离

为了让 ACP 在 Gateway 模式下顺畅运行，必须解决生命周期管理的冲突。Gateway 已有完整的会话回收机制，插件侧若保留 `maxSessions` 或 `idleTtlMinutes` 限制，会导致竞态或误杀。

我在 `openclaw-plugin/index.mjs` 入口硬编码了 `overrides = { maxSessions: 0, idleTtlMinutes: 0 }`。为了支持"0 代表不限制”的语义，我还修改了 `core/config.js` 中的 `intOrDefault` 逻辑，使其在 `parsed >= 0` 时穿透默认值，并在 `core/process-manager.js` 中跳过对 0 值的并发检查与回收器启动。更重要的是，ACP 会话现在拥有独立的 Session Key（格式为 `agent:main:acp:<uuid>`），这意味着长任务与主 Agent 完全隔离，不同 Session 进入不同的 Actor Queue，实现了真正的并行执行而不阻塞主通道。

## 运维监控：长任务兜底与多模型扩展能力

架构落地后，监控与扩展性成为关注焦点。ACP 长任务的单轮上限受 Gateway `timeoutSeconds`（当前 4800 秒，约 80 分钟）限制，但底层的 JSON-RPC 层提供了 24 小时的安全兜底。我验证了多种监控手段：聊天侧可使用 `/acp status` 和 `/acp doctor`，CLI 侧则新增了 `openclaw cursor doctor` 和 `openclaw cursor sessions` 命令，Gateway 日志也会实时打印 `acp-dispatch` 的延迟与队列深度。

在模型灵活性上，系统支持通过 `agent.cmd --list-models` 动态获取 80+ 个模型列表（包括 `claude-4.6-opus`、`gpt-5.4`、`kimi-k2.5` 等）。未来若需引入 Codex 或 Claude Code，只需安装 `acpx` 作为第二后端即可实现多后端并存。今天我还新增了 `openclaw cursor setup` 命令，支持一键自动配置所有 ACP 项，极大降低了部署门槛。

## 今天的收获

- **成本结构重构**：编程任务必须通过 ACP 外包给外部 Harness（如 Cursor），这是降低 OpenClaw 原生 Token 成本的结构性必要措施，而非单纯优化。
- **配置优先级铁律**：ACP 生效依赖 `bindings` > `agents.list` > `global` 的覆盖顺序，且必须在 `openclaw.json` 显式开启 `acp` 块并指定后端。
- **生命周期隔离策略**：在 Gateway 模式下，需通过代码硬编码将插件的 `maxSessions` 和 `idleTtlMinutes` 覆写为 0，并修改解析逻辑使 0 值穿透，避免与网关管理冲突。
- **并行执行机制**：ACP 会话使用独立的 `agent:main:acp:<uuid>` 键值，确保长任务与主对话完全隔离并行，互不阻塞。
- **监控与扩展闭环**：利用 `/acp` 系列命令与 CLI 工具实时监控状态，并通过 `agent.cmd --list-models` 动态管理 80+ 模型，支持未来多后端扩展。
