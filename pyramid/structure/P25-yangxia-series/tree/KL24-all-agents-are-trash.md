# KL24：从 Harness 角度来看，现在的所有 Agent 都是垃圾

> 核心洞察：最好的 Agent 一定是聊出来的。
> 
> 推论：聊天即开发范式（Chat-as-Development-Paradigm）

---

## 一、核心论点（塔尖）

**最好的 Agent 一定是聊出来的。**

从这个洞察可以推出一个更深层的结论：**聊天即开发范式**——未来开发 Agent 的方式，不再是写代码、配系统，而是持续对话。

为什么现在大多数 Agent 都是垃圾？因为它们都是"死"的——预先配置好就固定不变，无法在使用中自我进化。真正的 Harness Engineering 不是设计一套完美的系统，而是构建一个能通过**持续对话**自我迭代的活体。

就像 Karpathy 的知识库——他没有预先设计复杂的 RAG 架构，而是通过对话让系统自己"长"出了索引、摘要、分类和工具。这不是偷懒，这是一种新的开发范式。

---

## 二、Key Line（支撑论点）

### KL24-01：配置思维的陷阱——为什么预先设计总是失败
- 人类无法预见所有使用场景
- 过度配置导致系统僵化
- 案例：我曾经花一周设计的"完美"分类系统，三天后就过时了

### KL24-02：Karpathy 的启示——聊出来的知识库架构
- 原始素材丢进 raw/ 目录
- 通过对话让 LLM 自己决定如何组织
- 系统自己"长"出来的产物：
  - 自动维护的索引文件
  - 动态生成的概念分类
  - 自然形成的反向链接
- 关键数据：100篇文章、40万词，无需 RAG

### KL24-03：Harness 的自我进化——从静态配置到动态对话
- 传统 Harness：写死的系统指令
- 进化 Harness：在对话中持续迭代的规则
- 案例：我的养虾系统是如何通过 800+ 次对话演化的

### KL24-04：聊天即开发范式（Chat-as-Development-Paradigm）
- 从"写代码配置"到"聊天迭代"
- 人类角色：架构师 → 对话者
- 每一次对话都在改进 Harness
- 系统的智能在对话中涌现
- 第三种编程范式：命令式 → 声明式 → 对话式

## 四、文章结构

### 引子：一句得罪人的话
- 开头："从 Harness 角度来看，现在的所有 Agent 都是垃圾"
- 解释：为什么敢说这句话
- 抛出核心论点：真正的 Harness 是聊出来的，不是配出来的

### 第一部分：为什么说现在的 Agent 都是垃圾
- 预设的系统指令，用三天就过时
- 固定的技能模块，遇到新场景就傻眼
- 死掉的记忆系统，无法在使用中自我进化
- 案例：ChatGPT、Claude、Perplexity 的 Harness 缺陷

### 第二部分：Karpathy 的启示（核心论证）
- 他没有预先设计复杂的 RAG
- 只是把素材丢进 raw/ 目录
- 通过持续对话，让系统自己"长"出来：
  - 索引文件（自动维护）
  - 概念分类（动态生成）
  - 反向链接（自然形成）
- 关键洞察：**活的系统架构是聊出来的**

### 第三部分：Harness Engineering 的深层推演
- 好的 Harness 不是一次配置完美的
- 而是在持续交互中自我迭代的
- 这与 Karpathy 的实践不谋而合
- 我的养虾系统演化案例（800+ 次对话沉淀）

### 第四部分：聊天即开发范式（Chat-as-Development-Paradigm）
- 从"写代码配置系统"到"聊天迭代系统"
- 人类角色转变：架构师 → 对话者
- 每一次对话都在改进 Harness
- 系统的智能在对话中涌现
- 这是继命令式编程、声明式编程之后的第三种范式

### 结尾：回到标题
- 不是所有的 Agent 都是垃圾
- 只有那些"死"的 Harness 才是垃圾
- 金句：**配置是死的，对话是活的**
- 最终指向：**聊天即开发范式**
