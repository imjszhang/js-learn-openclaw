## 支撑材料

### 选题背景
- **来源**：用户想追踪 Karpathy/a16z/OpenClaw 动态，但手动刷 X 太浪费时间
- **痛点**：信息过载与时区错位导致漏看重要信息
- **方案**：开发 x-monitor 技能，定时监控 + 智能摘要 + 飞书通知
- **成果**：已监控 @karpathy @openclaw @a16z @moltbook @steipete 等账号，累计检查 28 次，发现 14 条推文

### 核心技术点
1. **JS-Eyes 登录态复用**：通过浏览器扩展解决 X.com 反爬
2. **双重去重策略**：内容哈希 + ID 去重
3. **成本可控**：月 Token 消耗约 1.3M，成本 $0.3-0.5
4. **Cron 自动化**：每小时检查一次，自动推送飞书

### 踩坑记录
- **坑 1**：`x_get_profile` 在 isolated session（Cron 环境）中不存在 → 改用 `js_eyes_open_url` + `js_eyes_execute_script`
- **坑 2**：X.com DOM 频繁变动 → 优先使用 data-testid 稳定属性
- **坑 3**：Rate Limit 限制 → 控制频率 + 随机延迟
- **坑 4**：登录态过期 → 检测状态 + 人工提醒

### 关键数据
- 监控账号：5 个（karpathy, openclaw, a16z, moltbook, steipete）
- 检查频率：每小时 1 次
- 累计检查：28 次
- 累计发现：14 条推文
- 信噪比：约 50%
- 月成本：$0.3-0.5

## 逻辑结构

### 引子：为什么我要花 10 天做一个监控工具？
- 场景：想追踪 Karpathy/a16z/OpenClaw 动态，但手动刷太浪费时间
- 调研：现有方案要么贵（付费监控工具），要么被封（官方 API 限制）
- 决定：用 OpenClaw + JS-Eyes 自己做一个
- 悬念：10 天，从 0 到上架 ClawHub

### Day 1-2：需求与选型
**Day 1：需求觉醒**
- 明确监控目标：5 个技术大佬账号
- 定义 MVP 功能：抓取 → 去重 → 摘要 → 推送
- 确定成功标准：不漏掉重要推文，不打扰非重要推文

**Day 2：技术选型**
- 方案 A：官方 API（贵、被封、限制多）→ 放弃
- 方案 B：第三方库（不稳定、维护差）→ 放弃
- 方案 C：JS-Eyes 浏览器扩展（复用登录态、稳定、可控）→ 选定
- 关键决策：用浏览器自动化解决反爬问题

### Day 3-4：架构与原型
**Day 3：架构设计**
- 数据流设计：X.com 时间线 → JS-Eyes 扩展 → WebSocket → OpenClaw Gateway → Agent/Cron → 飞书推送
- 模块划分：抓取模块、去重模块、摘要模块、通知模块
- 状态管理：状态文件记录已推送推文 ID

**Day 4：核心开发**
- 实现 X.com 页面打开（`js_eyes_open_url`）
- 实现推文提取（`js_eyes_execute_script`）
- 实现数据解析（提取内容、时间、互动数）
- 产出：可运行的命令行原型

### Day 5-6：去重与摘要
**Day 5：去重策略**
- 问题：X.com 时间线会重复显示同一推文
- 方案 1：ID 去重（记录已推送推文 ID）
- 方案 2：内容哈希去重（防止内容相同但 ID 不同的情况）
- 双重保险：ID + 哈希双重校验

**Day 6：摘要生成**
- 接入 AI 模型（Kimi K2）生成中文摘要
- 评分机制：根据关键词、互动数、作者重要性打分
- 筛选逻辑：只推送高价值推文

### Day 7-8：通知与自动化
**Day 7：通知推送**
- 接入飞书 webhook
- 设计消息模板（标题 + 摘要 + 原文链接 + 互动数据）
- 测试推送效果

**Day 8：Cron 自动化**
- 配置 Cron 定时任务（每小时执行一次）
- 状态文件管理（记录检查时间、最新推文 ID）
- 首次运行测试

### Day 9：测试与优化
**遇到的坑与解决方案**
- **坑 1**：Cron 环境 `x_get_profile` 不存在 → 改用 JS-Eyes
- **坑 2**：X.com DOM 变动导致抓取失败 → 使用 data-testid 稳定属性
- **坑 3**：Rate Limit 限制 → 控制频率 + 随机延迟
- **坑 4**：登录态过期 → 检测状态 + 发送人工提醒

**稳定性优化**
- 异常捕获与重试机制
- 日志记录与问题排查
- 边界情况处理（空时间线、网络错误）

### Day 10：打包与发布
**编写 SKILL.md**
- 技能描述与适用场景
- 提供的工具列表与使用方法
- 配置参数说明
- 使用示例

**ClawHub 上架**
- 打包技能目录
- 提交审核
- 审核反馈：需说明 X.com 数据获取合规性
- 补充合规说明后通过审核

**发布后数据**
- 上架 7 天，安装量：待补充
- 用户反馈：待补充
- 版本迭代：已更新 2 个版本

### 结尾：10 天开发的核心收获

**技术层面**
- 学会了用 JS-Eyes 解决登录态难题
- 掌握了浏览器自动化在 Agent 中的应用
- 理解了 Cron 定时任务的配置与调试

**产品层面**
- 明白了「一个技能 = 一个具体问题」的设计哲学
- 体验了从「给自己用」到「给别人用」的转变
- 理解了开源协议、文档编写、审核流程

**认知层面**
- 技能开发不是「写代码」，而是「封装能力」
- 好的 Skill 要有明确边界、清晰接口、完整文档
- 社区共建是 OpenClaw 生态的核心

### 给新手的建议

**开始之前**
1. 先想清楚解决什么问题，不要从技术出发
2. 调研现有方案，明确差异化价值
3. 定义 MVP，不要追求大而全

**开发过程中**
1. 先让原型跑起来，再优化细节
2. 记录踩坑过程，这是后续文档的素材
3. 多测试边界情况

**发布之前**
1. 文档比代码更重要
2. 从用户角度写使用说明
3. 准备好应对审核反馈
