## Key Line 树

### KL01 - Hermes 火了，龙虾要被取代了吗？

**功能：开头反转（热点新闻 + 反问）**

**设计**：
Hermes Agent 4 万星。
文章说它是"Agent 操作系统"。
评论区：该不该换？
我的回答：不用换。
不是因为龙虾比 Hermes 强。
而是因为——Hermes 做的每一件事，都在证明你养龙虾的方向是对的。

**开头类型**：热点新闻 + 反问

---

### KL02 - Hermes 到底新在哪？

**功能：客观拆解（不贬低、不吹捧）**

**支撑**：
- 7 层架构：多入口 → 核心循环 → Prompt 组装 → 工具/Skill → 记忆 → 上下文治理 → 模型层
- 核心突破：把 Agent 从"对话"升级为"长期运行的系统"
- 控制循环：State → Decide → Execute → Update → Stop
- 记忆双线：语义记忆（记事）+ 程序性记忆（做事方法）
- 上下文治理：系统强约束，不靠模型自觉
- 子 Agent 并行：独立上下文，多任务调度
- **一句话总结**：Hermes 在认真做 Agent 的系统工程

---

### KL03 - 但这些东西，龙虾不早就在做了吗？

**功能：认知反转**

| Hermes 的"新" | 龙虾的做法 |
|---|---|
| 控制循环 | heartbeat + cron 调度，定时检查、自动执行 |
| Prompt 分层 | AGENTS.md（稳态）+ 对话输入（瞬时） |
| 语义记忆 | MEMORY.md + Prism 知识库 |
| 程序性记忆（Skill） | skills/ 目录，每个 Skill 一套 SKILL.md |
| 上下文治理 | 工作区文件系统天然隔离，不堆在一个上下文里 |
| 子 Agent 并行 | ACP 子代理（Cursor/Claude Code），独立会话 |
| 事件驱动 Gateway | 飞书/微信/Telegram 多渠道消息接入 |
| 最大迭代数控制 | 系统级 timeout 配置 |

**关键一句话**：
> Hermes 用 7 层代码实现的事，OpenClaw 用几个文件 + 一个调度器就解决了。
>
> 不是因为 OpenClaw 更高级。
>
> 是因为文件系统，本来就是最好的操作系统。

---

### KL04 - Hermes 真正证明的，不是"它强"，而是"我们对"

**功能：核心论证**

**支撑**：
- 2 年前，OpenClaw 走这条路时，大多数人觉得"用文件管理 Agent"太原始
- 现在，4 万星的 Hermes 也在做同样的事：记忆沉淀、Skill 机制、上下文治理
- **区别只是实现方式**：Hermes 用代码，OpenClaw 用文件
- **但底层逻辑完全一致**：Agent 不应该是一次性对话，而应该是长期运行的系统
- Hermes 的火，不是因为它是新框架——是因为它验证了一个被 OpenClaw 走了很久的方向

---

### KL05 - 所以不用换。继续养。

**功能：结论 + 个人实证**

**支撑**：
- 26 篇养虾日记，底层全是同一套东西：工作区里那几个文件的排列组合
- 如果 Hermes 证明了什么——那就是"把判断变成系统"这条路是对的
- 你不需要换框架。你需要的是把龙虾养好。
- 从第一个 skill 开始，从第一条记忆开始，从第一个自动化任务开始
- 框架会变，系统不会

**结尾金句**：
> Hermes 没有取代龙虾。
> 
> 它只是证明了——你从一开始就选对了路。
