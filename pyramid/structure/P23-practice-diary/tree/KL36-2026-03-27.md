# KL56: ACP 架构通过将编码任务外包给外部 Harness，从结构上解决了 OpenClaw 原生编程的高 Token 成本难题

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 56.1: 将高消耗的读写文件操作剥离至外部 ACP 运行时，是降低 Chat 模型上下文成本并实现长任务并行的必要架构演进
- 逻辑顺序：结构
- 引用 atoms: AC-01, AC-02, AC-03, AC-17
- 引用 groups: G81

OpenClaw 原生编程成本高昂的根源在于文件读写等操作大量消耗 Token，而 ACP 协议允许 Gateway 将这些任务外包给外部 Harness（如 Cursor），由外部工具承担编码成本。这并非可选优化，而是从成本角度使用 OpenClaw 的结构性必要措施，实现了主对话与编码执行的物理隔离。

### 56.2: 实施严格的配置优先级与生效机制，确保 ACP 后端在 Gateway 模式下正确激活且无生命周期冲突
- 逻辑顺序：结构
- 引用 atoms: AC-04, AC-05, AC-07, AC-08, AC-09
- 引用 groups: G81

ACP 配置遵循"bindings > agents.list > 全局”的覆盖顺序，需在 `openclaw.json` 中显式开启 `acp` 块并指定 `backend: "cursor"` 才能生效。同时，插件元数据需声明配置 Schema，并移除 Gateway 模式下无效的限制字段，以确保外部运行时能被正确调用。

### 56.3: 通过代码级覆写禁用本地会话限制，并利用独立 Session Key 实现与主 Agent 的完全隔离并行
- 逻辑顺序：结构
- 引用 atoms: AC-06, AC-10, AC-11, AC-12, AC-13
- 引用 groups: G81

在 Gateway 模式下，必须在插件入口硬编码覆写 `maxSessions` 和 `idleTtlMinutes` 为 0，并修改解析逻辑使 0 值穿透默认处理，从而避免与 Gateway 的生命周期管理冲突。ACP 会话拥有独立的 `agent:main:acp:<uuid>` 键值，确保长任务与主对话完全隔离且可并行执行。

### 56.4: 建立完善的运行时监控与多后端扩展机制，保障长任务执行的安全兜底与模型灵活性
- 逻辑顺序：结构
- 引用 atoms: AC-14, AC-15, AC-16, AC-18, AC-19
- 引用 groups: G81

ACP 长任务受 Gateway `timeoutSeconds` 限制，但 JSON-RPC 层提供 24 小时安全兜底，可通过 `/acp status` 或 `openclaw cursor doctor` 实时监控状态。系统支持通过 `agent.cmd --list-models` 动态获取 80+ 模型列表，并允许安装 `acpx` 作为第二后端以实现 Codex 或 Claude Code 的多后端并存方案。
